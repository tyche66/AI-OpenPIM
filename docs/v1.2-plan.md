## 5. 工作包详述

### 5.1 CI 与发布门禁升级（P0，M2）

目标：把本地验收门禁真正放进 CI，阻止测试不全的版本发布。

CI 必须执行（`.github/workflows/ci.yml`）：

- `backend-lint`：`pip install ruff && ruff check app/`
- `backend-test`：安装 `requirements.txt` 与测试依赖，`python -m pytest -q`
- `backend-compileall`：`python -m compileall app -q`
- `frontend-typecheck`：`npm ci && npx vue-tsc --noEmit`
- `frontend-lint`：`npx eslint . --max-warnings=0`（不做 `--fix`）
- `frontend-unit`：`npx vitest run`
- `frontend-build`：`npm run build`，上传 `frontend/dist` artifact
- `compose-validation`：`docker compose -f docker-compose.yml config -q` 与 `docker-compose.dev.yml`
- `migration-upgrade`：在一个全新临时 PG16 容器中执行 `alembic upgrade head`，必须从空库升到 head，禁止修改 `0009` 及更早迁移
- `dependency-audit`：`pip-audit`（high/critical 禁止发布，生成 SARIM 或文本报告）；`npm audit --omit=dev --audit-level=high`
- 产物上传：后端测试报告、前端测试报告、构建产物、审计报告作为 artifact

发布门禁（`RELEASE_GATE.md`）：

- 任一 lint/typecheck/test/build/Compose/migration/审计条目失败即 NO-GO
- 不允许“测试通过但运维未验收”的条件式发布
- migration 链只可新增，不得变更历史；新 schema 必须新增 `0010` 及以后的迁移
- 依赖漏洞门禁：high/critical 禁入发布
- 最终 GO 必须同时满足功能、CI、浏览器、恢复、性能、安全六大门禁

当前缺口（取自 `AI-OpenPIM/.github/workflows/ci.yml:9-58`）：CI 仅做后端 Ruff + AST 与前端 build，明显弱于本地验收标准，是 V1.2 的首要工程缺口。

### 5.2 运维可观测性（P0，M3）

目标：能发现服务异常、备份失败、容量异常和关键业务错误，不建设完整可观测平台。

健康探针（`backend/app/api/v1/health.py` 与 `backend/app/main.py:147-190`）：

- 保留 `/health/live`（liveness，仅进程存活）
- 扩展 `/health/ready`（readiness）：
  - 保留 db / redis / minio 检查
  - 增加 Gotenberg 与 OCR 可用性（关闭时显示 `disabled`/`none`，不误判故障）
  - 输出 `app_version`、`migration_head`、`started_at`，不输出任何秘密
- readiness 不应因低风险告警直接失效；严重容量不足才进入 NOT OK

指标 `/metrics`（Prometheus 文本格式，`backend/app/main.py` 新增路由）：

- `pim_http_requests_total{method,route,status}` 计数器
- `pim_http_request_duration_seconds` 直方图（buckets：0.01…5）
- `pim_db_pool_in_use` / `pim_db_pool_available`
- `pim_ai_requests_total{adapter,outcome=success|fail|timeout|ratelimit}`
- `pim_ocr_requests_total{adapter,outcome}`
- `pim_backup_last_success_timestamp` / `pim_backup_last_failure_timestamp` / `pim_backup_status{batch}`
- `pim_volume_free_bytes{target}` / `pim_volume_threshold_bytes{target}`

日志脱敏扫描：审计 `body_summary`（`backend/app/middleware/audit.py:171`）等记录点，确认不会落盘 JWT、密码、AI Key、MinIO 凭据或请求正文中的敏感字段。

内部运维检查端点 `/ops/status`（管理员鉴权）：

- 服务状态
- 最后备份时间与状态
- 当前数据库版本与 migration head
- `AI_ADAPTER` / `OCR_ADAPTER` 开关状态
- 最近 24 小时 5xx 数量
- 最近审计事件摘要（不含 `request_body`）

### 5.3 备份自动化与恢复复验（P0，M3）

统一备份约定（`scripts/db_backup.sh` 与 `scripts/minio_backup.sh`）：

- PostgreSQL 与 MinIO 共用一个备份批次 ID（`<UTC ISO8601>-<short uuid>`）
- 每份备份生成 SHA-256
- 生成 `backups/<batch_id>/manifest.json`：批次 ID、UTC 时间、数据库版本、migration head、PG 文件名与大小、MinIO 文件名与大小、对象数量、SHA-256
- 使用临时文件写入，成功后 `rename(2)` 原子改名，避免半成品被视为有效备份
- 数据库成功但 MinIO 失败时，整个批次标记 `incomplete`，并写 `backups/<batch_id>/status.json`

调度样例（`scripts/backup.timer` + `scripts/backup.service` 或 `scripts/backup.cron`，二选一推荐）：

- 默认每日执行，保留不少于 7 个日备份
- 配置只引用 env 文件，不把密码写入命令行

恢复验收（`scripts/restore_drill.sh`）：

- PG 恢复至独立 PG16 临时实例（不挂载生产卷）
- MinIO 恢复至独立临时 bucket
- 核对产品、分类、方案、报价、说明书、chunk、附件；对 MinIO 对象逐文件校验 hash
- 记录实测 RPO / RTO；目标 RPO ≤ 24h、RTO ≤ 4h

### 5.6 性能规模与并发基线（P1，M4）

- `1x`：当前试点规模（约 13 产品）

并发正确性验证：

- `ShareToken` 不超过最大访问次数（已实现：`backend/app/api/v1/share_token.py:240-263`，需新增并发压测用例）
- JWT refresh 不发生旧 token 重放（`backend/app/api/v1/auth.py:84-115`，需新增用例）

- 同一说明书重复处理不产生重复 chunk
- 并发修改返回明确冲突，不静默覆盖

性能优化原则（不预先优化）：

- 仅在 `EXPLAIN ANALYZE` 有证据后才加索引
- OFFSET 尚未成为瓶颈前不全部切换 keyset
- 若 100x 数据下出现深分页问题，再为产品列表引入 keyset pagination
- 不通过关闭 RBAC、脱敏或审计换取性能

## 6. 发布验收标准

### 6.1 功能与数据

- 13 条现有试点产品可继续浏览、筛选、导出
- 数据质量问题可筛选、统计、导出
- 待核价产品不会显示虚假价格
- `purchaser` / `sales` / `viewer` 字段权限无回归
- 操作日志筛选、分页、时间范围可用
- 非管理员不能访问审计数据

### 6.2 工程质量

- 后端 Ruff、`compileall`、完整 pytest 全绿
- 前端 `vue-tsc`、ESLint、Vitest、production build 全绿
- Desktop Chromium 与 Mobile Chrome 核心链路全绿
- CI 执行完整门禁，不再只做 lint 和 build
- 空 PG16 数据库可从 `0001` 升级到新 migration head
- 不修改历史 migration
- 依赖漏洞：high/critical 禁入发布

### 6.3 运行与恢复

- 生产 Compose 全服务健康
- HTTP 跳转 HTTPS，HTTPS readiness 返回 200
- PostgreSQL 和 MinIO 定时备份可执行
- 最近备份状态可查询
- 独立恢复演练通过
- RPO ≤ 24h，RTO ≤ 4h
- 不删除或修改现有 Docker volume

### 6.4 性能

- 产品列表：并发 20、200 请求、warmup 20，连续三轮最差稳态 P99 ≤ 200ms
- 产品详情 P99 ≤ 150ms
- 普通写操作 P99 ≤ 500ms
- 错误率 < 1%
- readiness ≤ 60 秒
- 冷连接与稳态结果分别记录
- 至少形成 1x、10x、100x 三档容量报告

### 6.5 安全

- `AI_ADAPTER=none` 与 `OCR_ADAPTER=none` 为最终默认状态
- 外部 AI Key 未进入 Git、文档、日志、命令参数、测试产物
- JWT、数据库密码、MinIO 密码秘密扫描无命中
- 审计 API 不返回请求正文
- 备份调度命令不包含明文密码
- AI、OCR、备份失败均 fail-closed，不生成伪成功数据

## 7. 发布策略

- 版本号使用 `v1.2.0`，不沿用日期型 MVP-RC 名称
- 从 `main` 建立 `release/v1.2.0`，功能冻结后只接受 P0/P1 修复
- RC 阶段至少一次全新环境部署 + 一次基于现有数据的升级部署
- 生产回归从 25 项扩到约 35-40 项，新增：备份状态、审计页面、数据质量、migration、容量检查
- 最终 GO 必须同时满足功能、CI、浏览器、恢复、性能、安全六大门禁
